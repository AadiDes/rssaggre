# RSS Feed Aggregator

A backend service written in Go that aggregates content from RSS feeds. Users register for an API key, add feeds to the system, and follow the ones they care about. A background worker continuously scrapes every registered feed on a schedule and stores new posts in PostgreSQL, so each user can pull a personalized stream of posts from everything they follow.

---

## About this project

This project was built by following Boot.dev's **"Learn Web Servers in Go"** course (also published on freeCodeCamp's YouTube channel). The core implementation follows the course structure.

I'm keeping this repo as a working reference for Go backend patterns — HTTP routing and middleware, API-key authentication, type-safe database access, and concurrent background work. Items under [Roadmap](#roadmap) are extensions I'm building beyond the course material.

---

## Features

- **User registration** with generated API keys
- **API-key authentication** via middleware on protected routes
- **Feed management** — register RSS feeds in the system
- **Follow / unfollow** — a many-to-many relationship between users and feeds
- **Concurrent background scraper** — fetches feeds in parallel on a fixed interval
- **Duplicate-safe post storage** — a unique constraint on post URLs prevents re-inserting the same item
- **Type-safe database layer** — all queries generated by `sqlc` from plain SQL
- **Versioned migrations** — schema managed with `goose`

## Tech stack

| Layer | Choice | Why |
|---|---|---|
| Language | **Go** | The core workload is I/O-bound, highly parallel network fetching — goroutines make that cheap and simple. Static typing and a single-binary deploy. |
| Router | **chi** | Lightweight and fully `net/http`-compatible, so no framework lock-in. Clean middleware and route grouping. |
| Database | **PostgreSQL** | The data is relational and includes a many-to-many relationship. Foreign keys, transactions, and real indexing. |
| DB access | **sqlc** | Write real SQL, get compile-time type-safe Go. No ORM magic, no hidden N+1 queries, no runtime reflection. |
| Migrations | **goose** | Versioned, ordered, reversible schema changes that are reproducible across environments. |

## Architecture

The service is two independent execution paths sharing one PostgreSQL database:

```
                    ┌─────────────────────────┐
   HTTP clients ───▶│   REST API (chi)        │
                    │   ├─ logging middleware │
                    │   ├─ auth middleware    │──┐
                    │   └─ handlers           │  │
                    └─────────────────────────┘  │
                                                 ▼
                                        ┌──────────────┐
                                        │  PostgreSQL  │
                                        └──────────────┘
                                                 ▲
                    ┌─────────────────────────┐  │
                    │  Background scraper     │──┘
   RSS feeds  ◀─────│  ├─ ticker (interval)   │
   (network)        │  ├─ goroutine per feed  │
                    │  └─ sync.WaitGroup      │
                    └─────────────────────────┘
```

- **The API** is synchronous. Request in, response out, short-lived.
- **The scraper** is asynchronous. A long-running goroutine that ticks on an interval, selects a batch of the least-recently-fetched feeds, fans out over goroutines to fetch them concurrently, and writes new posts.

The two halves never call each other — they communicate only through the database. Fetching feeds is slow and periodic, so it must never sit inside a request/response cycle.

### Scraper scheduling

Each tick selects the next *N* feeds ordered by `last_fetched_at ASC NULLS FIRST` — never-fetched feeds first, then the ones fetched longest ago. Each goroutine marks its feed as fetched *before* doing the work, so the next batch won't re-select it.

## Database schema

Four tables, in third normal form:

| Table | Purpose |
|---|---|
| `users` | A registered user. Holds the unique `api_key` used for authentication. |
| `feeds` | An RSS feed registered in the system. `last_fetched_at` drives scraper scheduling. |
| `feed_follows` | **Join table** resolving the many-to-many between users and feeds. Unique on `(user_id, feed_id)`. |
| `posts` | A single scraped item. Unique `url` prevents duplicate storage. |

**Relationships**

- A user can add many feeds — `users 1→∗ feeds`
- A user follows many feeds, and a feed has many followers — `users ∗→∗ feeds`, via `feed_follows`
- A feed has many posts — `feeds 1→∗ posts`

## Getting started

### Prerequisites

- [Go](https://go.dev/dl/) 1.21 or newer
- [PostgreSQL](https://www.postgresql.org/download/) 14 or newer (or Docker — see below)
- [goose](https://github.com/pressly/goose) — `go install github.com/pressly/goose/v3/cmd/goose@latest`
- [sqlc](https://sqlc.dev/) — only needed if you modify queries

### 1. Clone and install dependencies

```bash
git clone https://github.com/<your-username>/<repo-name>.git
cd <repo-name>
go mod download
```

### 2. Configure environment

Copy the example file and fill in your values:

```bash
cp .env.example .env
```

| Variable | Description | Example |
|---|---|---|
| `PORT` | Port the HTTP server listens on | `8080` |
| `DB_URL` | PostgreSQL connection string | `postgres://user:pass@localhost:5432/rssagg?sslmode=disable` |

> **Note:** `.env` is gitignored and must never be committed — it contains database credentials.

### 3. Run migrations

```bash
cd sql/schema
goose postgres "$DB_URL" up
cd ../..
```

### 4. Run the server

```bash
go build -o rssagg && ./rssagg
```

The server starts on `PORT` and the background scraper begins on its interval.

## API reference

All routes are prefixed with `/v1`. Protected routes require an API key:

```
Authorization: ApiKey <your-api-key>
```

| Method | Endpoint | Auth | Description |
|---|---|:---:|---|
| `GET` | `/v1/healthz` | — | Readiness check |
| `GET` | `/v1/err` | — | Returns an error, for testing error handling |
| `POST` | `/v1/users` | — | Create a user, returns the generated API key |
| `GET` | `/v1/users` | ✔ | Get the authenticated user |
| `POST` | `/v1/feeds` | ✔ | Add a feed and auto-follow it |
| `GET` | `/v1/feeds` | — | List all feeds in the system |
| `POST` | `/v1/feed_follows` | ✔ | Follow a feed |
| `GET` | `/v1/feed_follows` | ✔ | List the feeds the user follows |
| `DELETE` | `/v1/feed_follows/{feedFollowID}` | ✔ | Unfollow a feed |
| `GET` | `/v1/posts` | ✔ | Get posts from all followed feeds |

### Examples

**Create a user**

```bash
curl -X POST http://localhost:8080/v1/users \
  -H "Content-Type: application/json" \
  -d '{"name":"aadi"}'
```

```json
{
  "id": "a1b2c3d4-0000-0000-0000-000000000000",

  "name": "aadi",
  "api_key": "5f4dcc3b5aa765d61d8327deb882cf99..."

```

**Add a feed**

```bash
curl -X POST http://localhost:8080/v1/feeds \
  -H "Authorization: ApiKey <your-api-key>" \
  -H "Content-Type: application/json" \
```

**Get your posts**

```bash
curl http://localhost:8080/v1/posts \
  -H "Authorization: ApiKey <your-api-key>"
```

## Project structure

```

├── main.go                 # entry point, router setup, scraper launch
├── handler_*.go            # HTTP handlers, one file per resource

└── .env.example
```

> `internal/database/` is generated. After changing anything in `sql/queries/` or `sql/schema/`, regenerate with `sqlc generate`.

## Roadmap

Planned extensions beyond the base course implementation:

- [ ] **Cursor-based pagination** on `/v1/posts` — keyset pagination on `(published_at, id)` with a composite index, so paging stays fast at depth and stable while the scraper writes new rows
- [ ] **Test suite** — table-driven handler tests with `httptest`, plus `go test -race` to verify the scraper
- [ ] **Bounded scraper concurrency** — replace unbounded goroutine fan-out with a worker pool over a jobs channel
- [ ] **HTTP client timeouts** — prevent a hung feed from stalling a scrape batch
- [ ] **Retry with exponential backoff** on failed feed fetches
- [ ] **Rate limiting** — inbound per API key, outbound per feed host

### Known limitations

notes on the current state:

- Feed fetches have no timeout — a hung host could hold a goroutine and stall the batch
- The scraper is single-process; running multiple instances would duplicate work without `SELECT ... FOR UPDATE SKIP LOCKED` or a work queue
- No retry logic on failed fetches — a flaky feed simply waits for the next cycle
- API keys have no expiry, rotation, or scopes
- Observability is limited to standard logging
